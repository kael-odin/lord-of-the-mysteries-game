# 《雾中余烬》完整游戏重建设计

日期：2026-07-16
状态：已确认总体方向，等待书面规格复核
项目：`D:\lord-of-the-mysteries-game`
远端：`https://github.com/kael-odin/lord-of-the-mysteries-game`

## 1. 决策摘要

本项目将从一个约 40 分钟的《诡秘之主》文字冒险原型，建设为正史兼容、原创主角驱动、可完整发布的调查角色扮演游戏。

采用的核心方向是：

- 游戏不是原著逐章复述，也不是让原创主角取代克莱恩的 AU。
- 玩家扮演 1349 年廷根的一名本地原创调查者，不是穿越者，不拥有源堡权限。
- 原著事件构成时间线和世界规则，玩家负责一条不改写正史核心因果的原创案件线。
- 邓恩、伦纳德、老尼尔、罗珊、弗莱和克莱恩等正史人物只能在合理时间、地点和权限范围内出现。
- 五条途径必须改变调查、社交、战斗、扮演、代价和结局，而不只是更换数值与技能名称。
- 完整首发战役为六章，单路线目标时长 8 至 12 小时，五路线完整体验目标时长 25 至 40 小时。
- 小说全文仅作为本机研究资料，不进入 Git 历史、发布包或部署环境。

暂定公开名称为：

> 《雾中余烬：诡秘之主同人互动冒险》

公开页面必须明确标注“非官方、非商业粉丝作品”，不得暗示获得阅文、作者或动画制作方授权。

## 2. 现状证据

### 2.1 已有资产

当前原型具有可复用的基础：

- Next.js 16、React 19、TypeScript strict、Tailwind CSS 4。
- 97 个唯一剧情节点、160 条静态跳转引用，未发现重复节点或字面悬空跳转。
- 五条途径、六类敌人、九个战斗入口、十四次 D20 判定和五个结局。
- 生命、灵性、理智、属性、幸运、金钱、消化度、背包、条件、一次性选项和隐藏选项。
- PostgreSQL、Drizzle ORM、单槽云存档和最近玩家回响原型。

### 2.2 阻断发布的问题

当前原型不能作为完成版继续堆内容，原因包括：

- 所有背景图均引用不存在的 `/public/images` 资源。
- 正常路线无法达到 100% 消化度，因此序列 8 和升级技能不可达。
- 至少 23 个剧情 flag 被写入后从未读取，选择后果没有形成闭环。
- 不眠者、窥秘人和猎人的剧情被动与实际引擎不一致。
- 战斗与判定不进入存档，刷新可以回滚并重掷。
- 存档 API 无所有权验证、运行时数据校验、版本迁移或正确错误码。
- 数据库异常被伪装成成功，界面无条件显示“存档已同步”。
- 仓库无锁文件、迁移、环境示例、README、测试、CI 或可复现启动说明。
- 当前剧情让卡尔复制克莱恩的枪伤、笔记与灰雾因果，同时让莫雷蒂一家继续存在，连续性冲突无法修补。
- 当前统一“序列 8 晋升仪式”不符合低序列晋升依靠配方、材料与服药的设定。

这些问题决定了项目需要“保留技术原型中的可用部分，重建内容与状态架构”，而不是只做文案扩写。

## 3. 产品边界与完成定义

### 3.1 目标玩家

首要玩家是熟悉《诡秘之主》第一部廷根篇、重视设定一致性和选择后果的读者。次要玩家是喜欢调查、克苏鲁氛围和叙事 RPG，但不熟悉原著的新玩家。

游戏提供两种知识呈现：

- 新人模式：关键名词可打开无剧透解释，隐藏中后期概念。
- 读者模式：减少基础解释，显示更多设定关联和原著时间线注记。

两种模式只影响解释密度，不影响难度或内容奖励。

### 3.2 首发范围

首发版本聚焦廷根原创旁线，不承诺改编全书八卷，也不让玩家升至高序列。完整成长范围为普通人、序列 9 和序列 8。

这不是缩减“完整游戏”的定义。首发版必须拥有完整开端、升级、冲突、终局和余波，并能独立作为一部结束的游戏。贝克兰德、海上和后续区域属于可独立立项的扩展战役，不能成为首发故事残缺的借口。

### 3.3 “100% 完整”的可验证定义

首发版只有同时满足以下条件才可称为完整：

- 六章主线全部完成，没有占位文字、临时按钮或未接线功能。
- 五条途径均有完整获得过程、扮演规则、专属任务链、序列 8 配方与晋升、专属终局变化。
- 每条途径至少有一项调查能力、一项社交能力、一项危机能力和一个独有代价。
- 至少 12 个可解锁结局，包含成功、代价成功、组织分歧、退隐、死亡和失控类型。
- 所有公开宣传的功能都由自动化测试和人工游玩证据覆盖。
- 所有剧情节点、资源、条件、奖励、经济和晋升引用通过构建期验证。
- 新游戏、继续游戏、多个存档槽、删除、导入、导出和版本迁移均正常。
- 在判定、战斗、仪式和章节转换中刷新页面，能够恢复到同一确定状态。
- 320px、375px、768px 和桌面视口完成实机视觉验收。
- 键盘、屏幕阅读器、减少动态效果和文本对比度达到可发布标准。
- 不包含小说全文、官方图片、动画截图、未经授权音乐或大段原著原句。

## 4. 正史兼容规则

### 4.1 不可改动的因果

- 周明瑞成为克莱恩并通过转运仪式进入源堡。
- 韦尔奇、娜娅、安提哥努斯笔记、瑞尔·比伯和 2-049 依照原著发生。
- 克莱恩的途径选择、塔罗会建立及廷根核心终局不由玩家替代。
- 正史人物的已知能力、所属组织、知识范围、管辖范围和生死时间不得为了方便玩法而改写。
- 低序列晋升依靠配方、材料和服药；序列 5 起才通常出现晋升仪式。

### 4.2 可创作空间

- 原著未描述的普通市民、教会外围人员、黑市商人、警察、记者和小型组织。
- 不改变正史结果的低序列案件、3 级封印物、地方传说与失踪事件。
- 正史章节之间合理存在的值夜者日常委托、跨组织协作和善后行动。
- 玩家与原创人物的关系、原创人物生死、案件解决方式和原创封印物归宿。

### 4.3 知识分级

每个内容节点标记知识等级：

- `common`：普通市民可知。
- `occult`：神秘学爱好者或低序列非凡者可知。
- `church-confidential`：官方非凡组织内部资料。
- `late-spoiler`：中后期原著概念，首发主线不得直接解释。

条件系统依据玩家身份、组织信任、线索和知识等级决定可见文字，避免新手教程提前泄露序列 0、源质、锚或旧日等内容。

## 5. 战役结构

### 第一章：无名信件

玩家作为廷根本地调查者收到一封准确描述陌生人死亡方式的无名信件。第一起死亡发生后，玩家因职业背景、私人关系或自身被点名而进入案件。

目标：

- 完成人物背景、姓名、代词和三项基础能力配置。
- 学会对话、观察、时间、线索、物品和风险判定。
- 建立三个可追踪关系，并让第一次选择产生后续证据差异。

### 第二章：非凡之门

案件触及低序列非凡力量。玩家通过不同渠道获得五条途径之一，不使用“桌上摆五瓶魔药”的方式。

渠道：

- 占卜家：教会库存与功勋审查。
- 不眠者：黑夜教会正式路线。
- 收尸人：黑夜教会尸检与亡灵事务路线。
- 窥秘人：神秘学研究与跨组织配方来源。
- 猎人：地下渠道与外勤协作者路线，带来更高组织怀疑。

### 第三章：廷根夜巡

城市枢纽开放。玩家在有限时间内选择委托、训练、关系事件、购物、恢复和扮演行动。

目标：

- 建立“调查案件”和“扮演消化”互相竞争时间的压力。
- 让五条途径第一次获得明显不同的解决方式。
- 引入 3 级封印物的使用代价和保管规则。

### 第四章：钟楼失踪案

原创案件进入核心调查阶段。玩家需要组合证据链，选择通灵、占卜、仪式、跟踪、潜入、审讯或公开执法。

目标：

- 至少三种完整调查路径可以定位真相。
- 错误结论不会立即结束游戏，但会改变受害者、时间和终局难度。
- 线索不是简单收集数量，而是具有来源、可靠性、污染和相互印证关系。

### 第五章：绯红假面舞会

敌对小型组织利用社交活动推进仪式。玩家必须在身份伪装、关系保护、证据公开和隐秘行动间取舍。

目标：

- 关系、组织信任和此前保存的证人决定可用盟友。
- 五途径各有一段独占解决路径。
- 序列 8 配方和材料通过此前选择获得，不由系统免费发放。

### 第六章：黎明前的余烬

终局由准备质量、调查结论、关系、污染、消化度和封印物归属共同决定。可行方案包括封印、净化、破坏仪式、交易、撤离和牺牲，不强制连续三场战斗。

结局后展示案件真相、人物余波、未解线索、途径表现、选择回顾和解锁内容。

## 6. 五途径差异矩阵

### 占卜家

- 调查：占卜、灵视、反占卜痕迹、仪式魔法。
- 社交：用观察与预兆引导对话，但不能把占卜当作确定事实。
- 危机：危机直觉、闪避和准备型仪式。
- 代价：受高位干扰时获得误导启示，过度依赖会降低证据可靠性。
- 序列 8：小丑，强化协调、表情控制、危险直觉和纸牌武器。

### 不眠者

- 调查：夜间视野、长时间监视、梦境与精神痕迹。
- 社交：稳定他人情绪，抵抗威压和疲劳审讯。
- 危机：黑暗环境强化、安魂诗篇和精神防护。
- 代价：长期昼夜颠倒导致白天行动效率与关系事件受损。
- 序列 8：午夜诗人，强化安抚、精神攻击和夜间团队增益。

### 收尸人

- 调查：尸检、亡灵感知、通灵和死亡痕迹。
- 社交：与死者家属和墓园人员建立特殊渠道。
- 危机：抵抗亡灵、寒冷和腐败，安抚或约束低阶死灵。
- 代价：死亡气息累积，普通人关系更难维持，错误通灵增加污染。
- 序列 8：掘墓人，强化身体素质、地底行动和死灵对抗。

### 窥秘人

- 调查：符文、材料、仪式结构、神秘学文献和污染来源。
- 社交：以知识交换情报，识别对方知识漏洞。
- 危机：提前准备符咒、药剂和反制仪式，不擅长无准备遭遇战。
- 代价：禁忌知识累积造成持续污染，错误阅读会永久改变可见文本。
- 序列 8：格斗学者，补足近身能力，但晋升来源必须符合组织与配方逻辑。

### 猎人

- 调查：追踪、地形、杀意、陷阱和敌方行动模式。
- 社交：挑衅、施压和识别敌意，容易制造额外冲突。
- 危机：伏击反制、环境陷阱和单体爆发。
- 代价：挑衅与攻击倾向提高组织怀疑，部分和平方案更难维持。
- 序列 8：挑衅者，强化情绪诱导、战场控制和风险收益。

## 7. 核心玩法系统

### 7.1 调查与证据

每条线索包含：来源、主题、可靠性、污染、关联人物、支持或反驳的假设。玩家在证据板上形成案件假设，系统依据证据组合解锁行动，而不是简单要求“收集三条线索”。

错误假设可以推进故事，但会消耗时间、降低组织信任或让敌人提前行动。

### 7.2 时间与城市

一天分为清晨、白天、黄昏、夜晚和深夜。移动、调查、恢复、训练与关系事件消耗时间。部分途径、地点和 NPC 只在特定时段开放。

时间压力用于制造取舍，不能导致玩家因一次普通探索永久卡死。主线关键节点前提供明确的最后准备提示。

### 7.3 判定

保留可理解的 D20 表现形式，但使用可保存的种子随机数。每次判定显示：

- 基础属性与途径修正。
- 证据、物品、关系和准备带来的优势或劣势。
- 成功概率与可能风险类别，不提前剧透具体结果。

失败必须推动新局面，不允许大量“失败后回原节点重试”。关键失败可通过消耗资源、接受代价或改变方案继续。

### 7.4 战斗、仪式与危机

战斗不是唯一危机解法。统一危机框架支持：

- 直接战斗。
- 逃离和追逐。
- 封印与仪式。
- 社交压制或拖延。
- 环境互动和队友协作。

敌人行动必须有可读预告。玩家可用证据、准备物和场景动作打断或改变敌方行动。战斗状态、随机种子和日志全部进入存档。

### 7.5 扮演与消化

消化度来自符合途径核心的具体行为，不来自普通击杀奖励。每次扮演事件记录采用了哪条守则、反馈是否真实、是否陷入“只扮演而忘记自己”的风险。

晋升序列 8 需要：

- 序列 9 消化达到要求。
- 获得正确配方。
- 收集主材料和辅助材料。
- 选择安全环境与护卫。
- 完成服药后的控制判定和途径专属场景。

### 7.6 关系与组织

关系不是单一好感度。每位关键人物至少包含信任、亲近和风险三个维度。组织记录功勋、怀疑和保密等级。

高信任不必然意味着同意玩家决定；人物依据自身职责和性格行动。

### 7.7 重玩

- 三个手动存档槽加一个自动存档槽。
- 章节检查点不覆盖常规存档。
- 结局图鉴隐藏未见结局名称，避免剧透。
- 二周目可快速跳过已读文字并显示新选项提示。
- 解锁内容限于背景、资料库、视觉主题和便捷功能，不提供破坏首轮平衡的永久数值。

## 8. 技术架构

### 8.1 保留与重建

保留 Next.js、React、TypeScript、Tailwind、Lucide、Drizzle 和 PostgreSQL。重建游戏域层、存档、安全、内容组织和主要界面。

现有代码先作为导入原型保留在 Git 历史中，再逐模块替换。不得边写新剧情边继续向 `story1.ts` 和 `story2.ts` 堆叠。

### 8.2 模块边界

建议目录职责：

```text
src/lib/game/
  model/          核心类型、ID、状态版本
  engine/         纯 reducer、动作、效果、条件、随机数
  content/        章节、案件、途径、物品、敌人、人物
  validation/     图、引用、平衡和内容规则验证
  persistence/    本地存档、云同步、迁移和冲突处理
  selectors/      UI 只读派生数据
```

UI 组件只派发动作和读取 selector，不直接修改状态。剧情内容只能声明条件与效果，不能包含任意业务代码。

### 8.3 统一状态

完整运行状态包含：

- `schemaVersion`、`contentVersion`、`runId`、`slotId`。
- 角色身份、途径、序列、属性、资源、消化和污染。
- 世界日期、时段、当前位置和剩余事件。
- 任务、线索、假设、关系、组织和知识权限。
- 背包、配方、材料、封印物及其负面效果。
- 当前剧情光标、待处理判定、危机状态、日志和结局记录。
- 可复现随机数种子与已消费步数。

所有状态变化通过 `dispatch(action, state, content, rng)` 返回新状态和领域事件。

### 8.4 内容格式

章节拆为独立 TypeScript 数据模块，使用 `satisfies` 和内容定义 helper 获得静态类型。运行和构建时再通过 schema 验证，覆盖：

- 唯一 ID。
- 所有节点、物品、人物、敌人、地点和任务引用。
- 所有路线从开局可达，并至少抵达一个终局。
- 非结局死路、无可选项节点和循环购买漏洞。
- 途径专属内容矩阵和晋升资源上界。

### 8.5 本地优先存档

本地存档是即时事实源，使用 IndexedDB 保存多个槽位。云端同步是可恢复副本，不能阻止离线游戏。

保存流程：

1. 领域动作完成。
2. 新状态原子写入本地槽位。
3. UI 显示本地已保存。
4. 后台提交带版本号的云同步。
5. 云端返回确认或冲突，UI 显示真实状态。

冲突采用显式版本比较，不使用最后写入静默覆盖。用户可以保留本地、采用云端或复制为新槽位。

### 8.6 服务端与数据库

服务端使用签名的匿名会话 cookie 标识所有权，客户端不再自行生成可用于授权的 `playerId`。

API 必须：

- 对请求和响应做运行时 schema 校验。
- 限制正文大小、名称长度和请求频率。
- 从 state 派生公开统计，不信任客户端冗余字段。
- 使用 4xx/5xx 表达真实错误。
- 对云存档使用乐观并发版本。
- 最近玩家回响改为明确选择加入，默认不公开名字和结局。

数据库包含会话、存档槽、运行摘要和迁移版本。游戏内容仍随应用发布，不存入数据库。

## 9. 体验与视觉

### 9.1 信息架构

首屏直接提供：继续最近旅程、新游戏、存档槽、资料库和设置。取消营销式长首页。

桌面主界面包含：

- 中央叙事与选择区。
- 可收起的证据板与任务日志。
- 城市地图和时间信息。
- 人物、途径、状态与行囊入口。

移动端使用固定底部工具栏和全屏抽屉，不让 HUD 多行挤压正文。

### 9.2 视觉语言

主色不是单一深蓝或金色。使用煤烟黑、旧纸白、氧化铜绿、煤气灯琥珀、暗红危险色和少量教会银。

场景以原创位图为主：廷根街道、黑荆棘公司、教堂、钟楼、舞会和终局地点。图片必须展示可辨认地点，不使用模糊氛围图替代内容。

卡片只用于重复条目、工具面板和模态框，不把每个页面区块都做成浮动卡片。

### 9.3 动效与声音

- 雾、灯光、低理智扭曲和受击反馈均可关闭。
- 遵循 `prefers-reduced-motion`。
- 环境音、音乐和音效分别控制音量并默认尊重浏览器自动播放限制。
- 关键结果不能只靠颜色、动画或声音表达。

### 9.4 可访问性

- 所有输入有可见 label。
- 状态条使用语义化进度信息。
- 判定、危机和抽屉具备对话框语义、焦点锁定和关闭后焦点恢复。
- 动态日志使用适当的 `aria-live`，避免重复朗读整页。
- 正文和操作文本达到可读字号与对比度。
- 所有核心流程可以只用键盘完成。

## 10. 错误处理

### 10.1 内容错误

缺失引用、不可达关键节点和无效条件在开发与 CI 中直接失败。正式版遇到内容版本不兼容时进入恢复界面，不能把玩家送到固定 hub 掩盖错误。

### 10.2 存档错误

- 每次迁移前保留原始快照。
- 无法迁移的存档进入隔离区并允许导出。
- 本地写入失败立即阻止继续覆盖并提示存储空间或浏览器权限问题。
- 云同步失败不删除本地进度，并提供重试。

### 10.3 网络与数据库错误

- 数据库不可用时，游戏进入明确的离线模式。
- 最近玩家回响失败不影响首屏与游戏。
- API 错误包含稳定错误码，不向客户端泄露堆栈、SQL 或环境变量。

### 10.4 运行时异常

关键界面具有错误边界，提供恢复最近本地快照、导出诊断信息和返回存档槽的选项。诊断信息不包含会话密钥、数据库地址或隐私内容。

## 11. 测试策略

### 11.1 引擎单元测试

覆盖：

- 效果、条件、属性边界和资源守恒。
- 判定概率、优势劣势和种子可复现性。
- 危机回合、状态效果、撤退、封印和胜负结算。
- 五途径能力、代价、扮演和晋升。
- 存档序列化、迁移和冲突处理。

### 11.2 内容验证

每次构建验证：

- ID 和引用完整性。
- 开局、五路线、晋升和结局可达性。
- 无意外死路和无限资源循环。
- 五途径内容矩阵满足首发要求。
- 所有图片、声音、字体和内容文件存在。
- 宣传数字与实际内容统计一致。

使用固定种子进行大量路径模拟，统计生命、灵性、污染、金钱、材料、消化和结局分布。模拟不是人工体验的替代，只负责发现数学上不可达或明显失衡。

### 11.3 API 与数据库测试

- 会话所有权和越权访问。
- 请求校验、大小限制和频率限制。
- 乐观并发冲突。
- 数据库中断、恢复和迁移。
- 回响选择加入和隐私字段。

### 11.4 浏览器验收

Playwright 覆盖：

- 新游戏、五途径选择、序列 8 晋升和主要结局。
- 多槽存档、刷新恢复、离线继续和云同步冲突。
- 判定与危机中刷新后不重掷。
- 320px、375px、768px 和宽屏布局。
- 键盘导航、焦点、减少动态效果和关键无障碍扫描。
- 所有静态资源无 404，场景图片非空白且构图可用。

### 11.5 人工游玩

每个阶段至少完成：

- 一次新玩家盲测。
- 一次原著读者设定审校。
- 一次五途径差异复盘。
- 一次从旧存档升级的回归测试。

首发候选版需要五条途径各完成一次端到端人工通关。

## 12. 版权与资料治理

- `docs/诡秘之主.txt` 永久列入 `.gitignore`，不得进入公开提交。
- 不使用官方站、动画、漫画、游戏或粉丝站图片作为发布素材。
- 游戏对白、旁白、案件和封印物说明重新创作。
- 标志性原句只在确有必要时短量引用，并在发布审查中逐项确认。
- 设定事实优先从本地原著核对，再用阅文官方站和可信资料补充交叉验证。
- 每个内容模块记录参考章节范围与剧透等级，但不复制原文。
- 商业化、众筹、广告、付费下载或品牌合作均在当前授权状态下禁止。
- 若准备公开大规模传播，应先通过阅文官方站提供的联系方式确认许可边界。

## 13. Git 与阶段交付

远端仓库目前为空。本地按以下方式建立历史：

1. 导入现有原型基线，排除小说全文和秘密文件。
2. 单独提交本设计规格。
3. 每个里程碑在独立分支实现，验证通过后合并并推送。
4. 提交信息描述行为变化，不混入无关格式化或生成文件。

阶段划分：

### M0：可复现与安全基线

- Git、远端、锁文件、README、环境示例、迁移和 CI。
- 本地优先存档骨架、正确错误状态和资源占位替换策略。
- 原型关键 P0/P1 问题形成自动化回归测试。

### M1：纯引擎与内容工具

- 统一状态、动作、效果、条件、随机数和 selector。
- 内容 schema、图验证、平衡模拟和存档迁移。
- 旧原型第一章迁入新引擎作为临时验证夹具，随后由新内容替换。

### M2：前两章纵切

- 新首屏、主游戏界面、证据板、地图、日志和移动端结构。
- 原创视觉基线和声音设置。
- 第一章、第二章和五途径获得流程完整可玩。

### M3：调查与成长主体

- 第三章、第四章、城市枢纽、时间、关系、组织和封印物。
- 五途径专属任务、扮演、材料和序列 8 晋升。

### M4：终局与重玩

- 第五章、第六章、约 12 个结局、余波、图鉴、二周目与资料库。
- 完整经济、污染、难度和路线差异调优。

### M5：发布验收

- 全路线浏览器与人工通关。
- 性能、无障碍、移动端、隐私、安全、备份和部署检查。
- 文案设定审校与版权素材审查。

## 14. 非目标

首发版不包含：

- 原著八卷的逐章改编。
- 玩家成为克莱恩、愚者或源堡拥有者。
- 高序列、神战、旧日或全世界地图。
- 实时多人、玩家交易、抽卡、广告或付费系统。
- 依赖生成式 AI 在运行时即时编写剧情。
- 通过无限随机事件伪装内容量。

这些边界保护首发战役的完整性、设定一致性和可验证质量。

## 15. 设计验收结论

本设计选择“正史兼容原创旁线”，保留现有 Web 技术栈，重建游戏域层和内容，采用本地优先存档、数据驱动剧情、可复现随机数、五途径差异化和六章完整战役。

进入实施前，书面规格必须由用户复核。复核通过后再编写逐任务实施计划，并从 M0 开始开发、验证和阶段推送。
